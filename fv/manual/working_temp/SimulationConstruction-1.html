Section 5 Simulation Configuration

5.1 Introduction

This chapter outlines simulation configuration commands and model selection including:

  • Coordinate reference frame, units and coordinate type
  • Solution include statements to enable the simulation of salinity, temperature and suspended sediment
  • Horizontal and vertical spatial order selection
  • Assignment of bed roughness and turbulent mixing models
  • Time commands and model timestepping

5.2 2D Hydrodynamics

5.2.1 Coordinate Reference Frame

TUFLOW FV requires that all input layers (mesh, GIS layers, boundary conditions etc.) imported or exported by TUFLOW FV must be in the same geographic projection. The model projection is initialised using the SHP Projection (if using QGIS of ArcGIS) or MI Projection command (if using Mapinfo).

The default output format for GIS check layers and GIS outputs depends on the projection setting as mentioned above. If more than one projection setting is specified, or to force the GIS output format, use the GIS Format command.

MJS nice to have here: To ensure TUFLOW FV is initialised correctly it is recommended to watch xx and yy… point at vids, or tutorial module 1.

Computations can be executed in cartesian or spherical (longitude latitude) coordinates via the Spherical command. Cartesian is the default in units of meters or feet depending on the specification of Units.

! GIS INTEGRATION
!_____________________________________
GIS Format == SHP
SHP Projection == ..\model\gis\projection.prj
Spherical == 1                                ! {0 Cartesian} | 1 Spherical}

Notes:

  1. If running in longitude/latitude coordinates, Spherical == 1 requires to be set regardless of the projection set by SHP Projection or MI Projection commands. If using spherical coordinates, the latitude is read directly from the model mesh and Coriolis deflection is spatially variable along the north-south axis of the model mesh.

  2. Note: If running with cartesian coordinates, Coriolis deflection is applied as a spatially constant value across the domain as a function of the specified Latitude. Care should be exercised to ensure the value of Latitude is representative of the model domain.

5.2.2 Spatial Order

The horizontal and vertical spatial scheme (the latter is for 3D models only) is specified as either 1st or 2nd second order using the Spatial Order command. For example:

! SIMULATION CONFIGURATION
!_____________________________________
Spatial Order == 1,1                    ! {1 1st Order Horizontal} | 2 2nd Order Horizontal, {1 1st Order Vertical} | 2 2nd Order Vertical

When considering the selection of 1st or 2nd order spatial schemes:

  • For rapidly varying flows that exhibit high spatial gradients in water level and velocity, such as tsunamis, dam breaks or flooding, a 2nd order solution is usually necessary and recommended. This is equally true when modelling high spatial gradients in density, or the concentration of suspended sediment, tracers or water quality constituents
  • 2nd order spatial accuracy will typically be required in the vertical direction when trying to resolve sharp stratification
  • In contrast, for 2D coastal modelling applications 1st order may be sufficient
  • Generally, it is recommended that initial model development is undertaken in 1st order with sensitivity testing completed using 2nd order. If a significant difference is observed, a 2nd order solution is likely required

For 2nd order models, users can optionally revert components of the solution to 1st order using the command Horizontal AlphaR for horizontal fluxes and Vertical AlphaR for vertical fluxes.

For example, to solve depth and velocity horizontal gradients in 1st order and 2nd order for plume/constituents, the following configuration can be implemented and is useful if 2nd order hydrodynamics are not required.

! SIMULATION CONFIGURATION
!_____________________________________
Spatial Order == 2,1                    ! {1} | 2 Horizontal, {1} | 2 Vertical 
Horizontal AlphaR == 0.0, 0.0, 1.0      ! 0.0 | {1.0} Depth, 0.0 | {1.0} Velocity, 0.0 | {1.0} Scalars

Notes:

  • If Spatial Order is set to 1 then the Horizontal AlphaR and Vertical AlphaR are ignored
  • Spatial order can also be adjusted for selected material types using the Spatial Reconstruction command within a Material Block (refer Section XX).

5.2.3 Units

The default Units for all inputs are Metric.

US Customary Units are supported for hydrodynamic only simulations and are not available when the Advection Dispersion, 3D, Heat, Water Quality, Sediment Transport or Particle Tracking Modules are enabled. As an alternative to US Customary Units, Imperial, and English are also accepted are treated identically.

! SIMULATION CONFIGURATION
!_____________________________________
Units == Metric                         ! {Metric} | US Customary | English | Imperial

This manual uses the Metric term for documentation purposes. For US Customary Units the metric term must be substituted for the equivalent US Customary unit as per Table 5.1.

Table 5.1: Model Units
Parameter Metric US Customary
Length m ft
Velocity m/s ft/s
Eddy Viscosity m2/s ft2/s
Bed Shear Stress (TAUB) N/m2 lbf/f2

5.2.4 Bed Roughness Model

The hydrodynamic bed boundary resistance is described using the Bottom Drag Model. The default model is that attributed to Manning, in which case a Manning’s “n” coefficient should be specified. An alternative bottom drag model assumes a log-law velocity profile and requires specification of a surface roughness length-scale, “ks” in meters.

! SIMULATION CONFIGURATION
!_____________________________________
Bottom Drag Model == Manning                        ! {Manning} | ks

A single bed surface roughness can be set globally, or the modeller can assign different roughness values to particular mesh cells within the model domain to spatially distribute the roughness using materials (refer Section 6.6).

For sediment transport simulations, the “ks” bottom drag model can be optionally coupled to the sediment transport roughness. This adjusts the bed roughness dynamically during a simulation as a function of the sediment characteristics. More details are provided in the Sediment Transport / Particle Tracking Module User Manual.

5.2.5 Time Commands

Two time formats are available and are set using the Time Format command:

  1. HOURS (the default) in units of decimal hours.

  2. ISODATE in the format dd/mm/yyyy HH:MM:SS.

The time format adopted must be consistent within all control files and model input time series.

The Start Time and End Time define the model simulation period.

Example using Time Format == HOURS

! TIME AND TIMESTEP COMMANDS
!_____________________________________
Time Format == HOURS                    ! {HOURS} | ISODATE
Start Time == 0.0                       ! (hrs)
End Time == 3.0                         ! (hrs)

Example using Time Format == ISODATE

! TIME AND TIMESTEP COMMANDS
!_____________________________________
Time Format == ISODATE                    ! {HOURS} | ISODATE
Start Time == 01/01/2024 00:00:00         ! (dd/mm/yyyy HH:MM:SS)
End Time == 01/04/2024 00:00:00           ! (dd/mm/yyyy HH:MM:SS)

If using time format == HOURS, the Reference Time default value is 0.0 hrs. For time format == ISODATE, the default Reference time is set to 01/01/1990 00:00:00.

5.2.6 Computational Timestep

5.2.6.1 TUFLOW FV’s Adaptive Timestep

By default, solution of the non-linear shallow water equations use an variable timestep where a stable time step must be bounded by the Courant-Friedrich-Lewy (CFL) criterion for the wave propagation and advective terms, and by the Peclet criterion for the diffusive terms. For more detail on these terms refer to the TUFLOW FV Science Manual.

The variable time step scheme is implemented to ensure that the CFL is maintained below the user specified CFL control number and Peclet criterion are satisfied at all points in the model with the largest possible time step. The minimum and maximum timestep limits are provided by the user via the Timestep Limits command.

5.2.6.2 Model Timestep Mode Splitting

Computations are completed using a mode splitting scheme, whereby “internal” and “external” components of the governing equations are updated using different ‘internal’ and ‘external’ timesteps selected by considering physical and numerical convergence and stability considerations.

The role of the two different timesteps is summarised as follows.

  • ‘External timestep’: The 2D depth-averaged shallow water wave speed is solved at this timestep.

  • ‘Internal timestep’: Internal waves, internal constituents with wave speeds are solved on this timestep, which is typically an order of magnitude higher than the external timestep.

An example of the calculated internal and external timesteps is shown in Figure 5.1. The internal timestep is highlighted by the yellow box, the external by the red.

Figure 5.1: Mode Split Computational Timestepping

Mode splitting can be disabled for 2D simulations using the Mode split command and this configuration can be more computationally efficient for fast, shallow flow scenarios where the “internal mode” and “external mode” timesteps are similarly restrictive. Currently, 3D simulations are only supported with mode splitting enabled.

Unless there are compelling reasons to turn mode splitting off it is recommended to run TUFLOW FV with mode splitting enabled. If unsure you can always run a sensitivity/benchmark tests and compare your results and runtimes to check if you gain any performance benefit.

5.2.8 Wetting And Drying Thresholds

MJS Trim and streamline this section..

TUFLOW FV simulates the wetting and drying of areas within the model domain, such as that observed on a gently sloping beach over a tidal cycle or over land during a flood, storm surge or tsunami event. Dry/wet depths defined by the user will often depend on the scale of the simulation. For full-scale or “real world” simulations, dry/wet depths are typically in the order of centimetres. For some laboratory-scale simulations, for example a dam break or wave run-up experiment, the user defined wet/dry depths may be in the order of millimetres.

Computationally, the drying value corresponds to a minimum depth below which the cell is dropped from calculations (subject to the status of surrounding cells), noting that mass is conserved. The wet value corresponds to a minimum depth below which cell momentum is set to zero, in order to avoid unphysical velocities at very low depths. The default wetting and drying values can be modified using the Cell Wet/Dry Depths command in the .fvc file.

5.2.9 Stability Limits

MJS Trim and streamline this section..

Major instabilities in models will usually manifest in shocks within the model that can result in unrealistically high water levels or velocities. However, as models differ in the range of expected values that they are trying to capture, there is no ‘rule’ for what water level or current velocity might indicate an instability. Therefore, users can specify upper limits that are relevant to their study and will cause the model to exit computations with an error if these are exceeded using the Stability limits command. Modellers are advised to set these limits to be as low as practicable to ensure that any real instabilities are ‘caught’ as soon as possible, but any small anomalous and acceptable spikes in water level or velocity do not cause a model exit. For typical ocean modelling in most parts of the world it might be appropriate to set these limits to a 10 m water elevation and a 5 m/s maximum current.

These limits will capture most of the highest expected water levels due to tide and storm surge effects as well as set an appropriate limit for tidal currents in an open area. If modellers are trying to capture extreme water levels or high currents through narrow choke points, then it is possible that these limits should be increased. Difficulties arise when investigating catchment models. In these situations, the topography may contain elevations above the maximum water level (if using a mean-sea-level datum for example). If any of these high cell elevations become wet due to rainfall inputs or upstream inflow boundary conditions, then this will instantly trigger a model exit. For these models it is important to set the maximum water level limit to be higher than the highest expected cell elevation that will get wet, plus any additional water depth expected in that cell. Traps for young players:

  • Water level limit needs to be higher than highest bathymetric value (at least, plus water depth)
  • Don’t use these as ways to get a model to run – they can tell you something about model instability

5.3 3D Hydrodynamics

5.3.1 Vertical Momentum Mixing

Vertical-mixing eddy-viscosity can be defined as a constant value or calculated using a parametric model or external scheme set by the Vertical mixing model command. Eddy-viscosity values for the constant and parametric models can be set globally via the Vertical mixing parameters command (if using an external scheme such as GOTM, the external model calculates the eddy viscosity (at a time increment defined by the turbulence update dt) and returns it to TUFLOW FV). The parametric model is based on a parabolic eddy-viscosity profile in the lower-half of the water column transitioning to a constant eddy-viscosity in the upper-half and applies the Munk & Anderson limiters in the case of stable stratification. For systems with strong vertical gradients the use of an external vertical turbulence mixing model is recommended. For parametric and external schemes eddy viscosity limits can be applied using the Global vertical eddy viscosity limits command. The global commands can be overwritten on a material basis using the vertical eddy viscosity limits command within a Material Block (refer Section 8.5).

5.4 Advection Dispersion/Diffusion

5.4.1 Solution Include Statements

By default, computations are executed in 2D depth averaged, hydrodynamic only mode. From this base level the user has the option to activate additional processes that can influence the simulation’s hydrodynamics such as salinity, temperature, atmospheric heat exchange, sediment and water quality. Include commands (refer Table 5.4) are typically added near the top of the .fvc file.

Table 5.4: Solution Include Statements
Command Details Notes
Include Salinity Optionally enable salinity modelling functionality with or without density coupling. Accessed via TUFLOW FV’s Advection Dispersion Module.
Include Temperature Optionally enable temperature modelling functionality with or without density coupling Accessed via TUFLOW FV’s Advection Dispersion Module.
Include Heat Optionally enable heat flux modelling between the water surface and atmosphere. Important for stratified environments or modelling with temperature enabled. Accessed via TUFLOW FV’s Advection Dispersion Module.
Include Sediment Optionally enable the modelling of cohesive or non-cohesive suspended sediments (with or without hydrodynamic density coupling) or bed load sediments. Sediment modelling is set via a separate TUFLOW FV Sediment Control File (.fvsed) [1] Accessed via TUFLOW FV’s Sediment Transport and Advection Dispersion Modules.

The user can choose to run a barotropic configuration (the default) or with the baroclinic pressure-gradient terms optionally activated to allow the hydrodynamic solution to respond to temperature, salinity and sediment induced density gradients using the Include Salinity, Include Temperature and/or Include Sediment commands. Typically baroclinic simulations will be conducted in 3D to allow for vertical density gradients, however 2D depth averaged baroclinic simulations are also supported.

These commands can be used together or in isolation. For example, setting Include Temperature == 1,1 and Include Salinity == 1,1 will result in both temperature and salinity being used for fluid density and baroclinic pressure gradient calculations, a common combination for coastal modelling. The influence of these terms may be significant in estuarine environments for example, where fluvial and marine waters meet, and stratification is known to occur.

An example .fvc file snippet from a coastal estuarine model is provided below that includes of salinity, temperature, sediment and atmospheric heat exchange. In this example salinity and temperature are considered in density calculations, while suspended sediment is not.

! SIMULATION CONFIGURATION
!_____________________________________
Include Salinity == 1,1               ! {0 Disabled} | 1 Enabled, {0 Barotropic} | 1 Baroclinic
Include Temperature == 1,1            ! {0 Disabled} | 1 Enabled, {0 Barotropic} | 1 Baroclinic
Include Sediment == 1,0               ! {0 Disabled} | 1 Enabled, {0 Barotropic} | 1 Baroclinic
Include Heat == 1                     ! {0 Disabled} | 1 Enabled

The effect of modelling salinity and temperature density effects is shown in the example 3D long section Figure 5.2. Fresh river water enters the system at chainage zero at the left of the plot. At the right the ocean tide is applied. The density effects at the mixing interface of river and ocean is clearly visible, with relatively buoyant river water moving over the salt water “wedge” below.

Figure 5.2: 3D Long Section of Salinity

5.4.2 Scalar Mixing

TUFLOW FV allows the horizontal scalar diffusivity to be specified as a constant value or calculated from a Smagorinsky or Elder model. The vertical scalar diffusivity may also be directly specified or calculated using parametric formulations which vary depending on the scalar type. For more complicated turbulence model algorithms an interface for linking with various external turbulence models has been implemented (e.g. GOTM, http://www.gotm.net/).

5.4.2.1 Horizontal Scalar Mixing

The Scalar mixing model command sets the horizontal-mixing scalar diffusivity as either a constant value or calculated using the Smagorinsky or Elder models. The respective coefficients for each approach are input via the Global horizontal scalar diffusivity command and can have minimum and maximum limits applied using the Global horizontal scalar diffusivity limits command. Spatially variable horizontal scalar diffusivity and horizonal scalar diffusivity limits can be applied within a Material Block (refer Section 8.5).

5.4.2.2 Vertical Scalar Mixing

The vertical-mixing scalar diffusivity can be defined as a constant value or can be calculated using a parametric or external model (see Vertical mixing model). Coefficients for each model are applied using the Vertical mixing parameters command. If using an external model, diffusivity is calculated on a user-defined time interval turbulence update dt. The parametric model is based on a parabolic eddy-viscosity profile in the lower-half of the water column transitioning to a constant eddy-viscosity in the upper-half and applies the Munk & Anderson limiters in the case of stable stratification. Upper and lower bound values can be specified globally using the Global vertical scalar diffusivity limits command and also on a material basis (refer vertical scalar diffusivity limits, and Section 8.5) which allows spatial variation.
